# AIコロニー箱庭

## 目的

`AIコロニー箱庭` は、AIが複数勢力を自律的に動かし、人間は基本的に放置して観察しながら、ときどき世界に介入して変化を楽しむための観察型シミュレーションである。

この案の主目的は、単にコロニー運営ゲームを作ることではない。

主目的は以下を満たすこと:

- 見ているだけで状況が分かる
- AIごとの性格差や判断差が見える
- 放置しても変化が続く
- 適度なランダム性がある
- 人がたまに触ると世界が崩れたり伸びたりする
- 長時間動かしても実用的である

## コア体験

このゲームの中心体験は「AIが作る社会や縄張りの変化を眺めること」である。

人は主に以下を楽しむ:

- どの勢力が広がるか
- どの勢力が資源不足で弱るか
- 争いが起こるか、膠着するか
- 性格の違うAIがどう振る舞うか
- 小さな介入で全体の均衡がどう変わるか

プレイヤーは常時操作する司令官ではなく、観察者兼たまに世界をかき回す存在である。

## 想定ジャンル

本案は以下の中間にある:

- RTS
- SLG
- 生態系シミュレーション
- ライフゲーム的観察
- オートバトル寄りの勢力シミュレーション

つまり「操作ゲーム」ではなく「観察対象としての勢力箱庭」を目指す。

## 最小世界モデル

最初の実装では、世界はシンプルでよい。

最低限必要な構成:

- マップ
- 勢力
- 資源
- 拠点
- ユニットまたは勢力パルス
- 競合
- ランダムイベント

### マップ

初期版では、複雑な地形生成は不要。

候補:

- グリッド
- ノード接続マップ
- 領域分割マップ

最初は視認性を優先し、どこが誰の支配圏かが一目で分かる形式にする。

### 勢力

最初は3勢力で十分。

各勢力は以下を持つ:

- 色
- 拠点
- 資源量
- 支配領域
- 性格パラメータ
- 行動優先度

### 資源

資源が不足しないと世界が平坦になるので、初期版から必須。

候補:

- food
- energy
- material

ただし最初から複雑にしすぎないため、初期版では `food` と `material` の2種でもよい。

### 拠点

各勢力は1つ以上の拠点を持つ。

拠点は:

- ユニット生成
- 資源蓄積
- 支配拡張の起点
- 攻撃対象

として機能する。

### ユニットまたは勢力パルス

初期版では本格ユニットAIでなくてもよい。

候補:

- 小型ユニットがマップを移動する
- 抽象化された勢力パルスが隣接領域へ伝播する

大事なのは「動いて見えること」であり、厳密な戦術AIではない。

## AI勢力の性格

このゲームでは、深いルールより先に「性格差」が重要である。

初期案:

- `hoarder`: 争いを避け、資源備蓄を優先する
- `raider`: 弱い相手や資源地帯を狙いやすい
- `builder`: 領土安定化と拠点強化を優先する
- `opportunist`: 周囲状況を見て勝てそうな場面だけ仕掛ける

勢力ごとの差を見せるため、以下を可視化するとよい:

- 勢力名
- 性格ラベル
- 現在の意図
- 直近の重要行動

## AIの判断対象

各勢力AIが毎サイクル考える対象は、最初は絞る。

最低限:

- どの領域を確保するか
- どこにユニットを送るか
- 資源採取を優先するか
- 拠点防衛を優先するか
- 拡張するか攻撃するか

この判断は、完全LLM任せでなくてもよい。

初期版では以下のどちらでも成立する:

- ルールベース + 重み
- ルールベース + ときどきLLM補正

## 観察しやすい画面構成

このゲームでは、可視化が主役である。

初期画面は以下の構成が望ましい。

### 1. メインビュー

表示内容:

- 地図
- 勢力色
- 拠点
- 動いているユニットやパルス
- 衝突地点
- 資源地帯

一目で「どこが伸びて、どこが危ないか」が分かる必要がある。

### 2. 観察HUD

表示内容:

- 各勢力の資源量
- 支配領域数
- 拠点数
- 性格
- 現在の意図
- 世界時間
- 現在のイベント

### 3. イベントログ

すべての細かい行動ではなく、観察価値の高いものだけを出す。

例:

- 新拠点設置
- 資源枯渇
- 勢力衝突
- 勢力壊滅
- 介入イベント発生

### 4. 介入UI

人が押せる介入ボタンを少数だけ置く。

候補:

- `storm`
- `resource_drop`
- `spawn_neutral`
- `invader_wave`

## 人の介入

このゲームにおける介入は、操作ではなく「世界改変」である。

介入の役割:

- 停滞を崩す
- 想定外の展開を作る
- AIの癖を観察しやすくする

初期版で十分な介入:

1. `storm`
特定領域を弱体化させる

2. `resource_drop`
特定領域に資源を投下する

3. `spawn_neutral`
中立勢力や野良拠点を増やす

4. `invader_wave`
外敵を流入させる

## ランダムイベント

放置観察を成立させるには、人が触らなくても変化が要る。

初期版の候補:

- 局所的な資源枯渇
- 豊作地帯の発生
- 災害で一時的に移動困難
- 拠点故障
- 中立地帯の発見

ポイントは、イベントが派手すぎて毎回ゲームを壊すのではなく、均衡を少しずつずらすこと。

## 最小実装仕様

最初の検証版では、以下だけ実装すればよい。

1. シンプルな2Dマップを生成する
2. 3勢力を生成する
3. 各勢力に性格を割り当てる
4. 資源点を配置する
5. 勢力が採取、拡張、衝突する
6. 領土色、拠点、移動、衝突を可視化する
7. 2種類以上のランダムイベントを入れる
8. 2種類以上の人間介入を入れる
9. 観察HUDと簡易イベントログを付ける

この段階では、勝利条件や複雑なストーリーは不要。

## 実装フェーズ案

### Phase 1: 世界の骨格

- マップ生成
- 勢力生成
- 領土表示
- 時間進行

### Phase 2: 行動ループ

- 資源採取
- 拡張
- 衝突
- 拠点維持

### Phase 3: 性格差

- 勢力ごとの重みづけ
- AI意図表示
- 行動傾向の差分観察

### Phase 4: 観察体験の強化

- イベントログ
- 危険地帯ハイライト
- 低負荷モード
- 速度変更

### Phase 5: 介入と長時間運用

- 世界イベント
- 人の介入ボタン
- run summary
- 複数試行比較

## 成功条件

この案の成功は「ゲームが成立すること」だけではない。

成功条件は以下:

- 5分以上放置しても画面の意味が追える
- 観察中に少なくとも数回は勢力関係の変化が起きる
- 勢力の性格差が見た目や結果に出る
- 人が1回介入すると、状況が目に見えて変わる
- ログを読まなくても、だいたい何が起きているか分かる

## 非目標

初期版では以下は不要:

- 複雑な外交システム
- 詳細な個体シミュレーション
- 手動RTSのような細かい命令
- 重い3D演出
- 完全に汎用的な設計

## 実装判断の指針

この案を実装するときは、以下を優先する:

- 厳密さより見やすさ
- 深さより差が見えること
- 完全自動化より観察価値
- 高密度処理より長時間回せること

もし迷ったら、「その要素は放置観察の面白さを増やすか」で判断する。
