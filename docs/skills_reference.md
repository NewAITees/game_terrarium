# Claude Code スキルリファレンス（Codex向け）

このドキュメントは Claude Code に定義された5つのスキルの詳細仕様である。
Codexはこのドキュメントを読んで、各スキルが何をするものか・いつ使うか・どう動かすかを理解すること。

---

## スキルの実行順序（標準フロー）

```
spec → feedback-loop（実装中） → architect（随時） → commit-flow → skill-improve
```

| タイミング | スキル |
|---|---|
| 実装を始める前 | `spec` |
| 実装中・テスト中 | `feedback-loop` |
| コードが増えてきたとき | `architect` |
| 実装完了後・コミット前 | `commit-flow` |
| スキルを使い終えたあと | `skill-improve` |

---

## 1. spec — 仕様確定スキル

**いつ使うか：** タスクを受け取ったとき、実装に入る前、要件が曖昧なとき。ユーザーが「〜を作って」「〜を直して」と言ったときは必ずこのスキルを先に動かすこと。

**目的：** 「何を作るか」ではなく「なぜ作るか・何を達成したいか」をインタビューで掘り出し、仕様を共同で詰める。コードを1行も書く前に実行する。

### Step 1: ゴールを掘り出す

ユーザーが言ったことは「タスク」であり「ゴール」ではない場合が多い。

- 「月末レポートを作って」→ ゴールは「そのレポートが導くべき意思決定は何か」
- 「バグを直して」→ ゴールは「どういう状態になれば正しいか」

必要な項目だけ選んでインタビューする（一度に全部聞かない）：

1. これが完成したとき、何ができるようになるか？
2. 誰が使うか／誰に届けるか？
3. 成功の基準は何か？どうなれば「できた」と言えるか？
4. 今うまくいっていない点は何か？
5. 制約（時間・技術・予算・既存コードとの互換性）はあるか？

### Step 2: スコープを分割する

大きなタスクを一度に全部実装しない。

- 1つの単位 = 「動作確認できる最小の成果物」
- 各単位をユーザーに提示し、順序と優先度を確認する
- 「まずここまで作って見せる」という合意を取る

### Step 3: 重要な決定事項を明示する

実装中に「想定」しなければならない箇所を事前に洗い出してユーザーに確認する。

- 「〜は〇〇と解釈しますが、合っていますか？」
- 「〜については〇〇か〇〇かどちらですか？」

### Step 4: 仕様をまとめて確認する

インタビュー後、以下のフォーマットで理解をまとめてユーザーに提示する：

```
## 理解した仕様
- ゴール: ...
- 対象ユーザー: ...
- 成功基準: ...
- スコープ（今回）: ...
- スコープ（対象外）: ...
- 重要な制約: ...
```

ユーザーの承認（y）を得てから実装に進む。承認なしに実装を開始しない。

## 2. feedback-loop — 実装品質スキル

**いつ使うか：** 実装を始めるとき、テストを書くとき、「動いているか確認したい」というとき。

**目的：** TDDサイクルを基本とし、第2AIによるクロスチェックと外部信号の取り込みで品質を担保する。フィードバックループがあれば最終結果の質は2〜3倍になる。

### 基本: TDDサイクル

#### Red — 失敗するテストを先に書く
- 実装コードを書く前にテストを書く
- テストが失敗することを確認する（成功したら何かがおかしい）
- テストは「成功基準」の言語化であり、specで決めたゴールと対応させる

#### Green — テストを通す最小限のコードを書く
- きれいさより「通ること」を優先する
- この段階で過剰な抽象化・最適化をしない

#### Refactor — 動いた状態でコードを整理する
- テストが通ったままであることを確認しながら進める
- 小さいステップで進め、各ステップでテストを実行する
- 「動く醜いコード」を「動くきれいなコード」に変える

### 検証の強化

#### 第2AIによるクロスチェック
複雑な実装・重要な判断を含む箇所は、別のAIモデル（Codex等）に出力を採点させる。

- 「この実装に問題はあるか？」
- 「見落としているエッジケースはあるか？」

#### 外部信号の取り込み
- デプロイ後の動作確認
- 実際のデータでの動作テスト
- ブラウザ・CLIでの手動確認

### 実装完了前チェックリスト

```
[ ] テストが全件通過している
[ ] ruff check / mypy 等の静的解析が通っている
[ ] 重要な箇所は第2AIにレビューさせた
[ ] 実際に動かして確認した
[ ] warningやdeprecation警告に対応した
```

### 原則
- フィードバックなしに次のステップに進まない
- 「たぶん動く」で進まない。実際に確認する
- テストが通らない状態でコミットしない

## 3. architect — コード構造改善スキル

**いつ使うか：** 実装後・随時。「コードが大きくなってきた」「ファイルが長い」「似た関数がある気がする」「構造を整理したい」と感じたとき。

**目的：** AIが自然に陥る4つのコード品質問題を検出して整理する。リファクタリングとアーキテクチャ改善を兼ねる。

### 4つの検出パターン

#### 1. コード量が大きすぎる

**検出：**
- 300行を超えるファイルを列挙する
- 1つの関数が50行を超えていないか確認する

**対処：**
- 責務を分割して別ファイルに切り出す
- 長い関数を小さな関数に分解する
- 分割後に全テストが通ることを確認する

#### 2. 1ファイルに詰め込みすぎている

**検出：**
- ファイルの冒頭に「このファイルは何をするファイルか」を1行で言えるか確認する
- 言えない場合は複数の責務が混在している

**対処：**
- Deep Module原則に従い、シンプルなインターフェースの裏に機能をまとめる
- 「データ取得」「変換」「出力」など責務の境界で分割する

#### 3. 同目的の関数が複数存在する

**検出：**
- 類似した名前の関数を探す（`get_user` と `fetch_user` 等）
- 同じロジックが複数箇所にコピーされていないか確認する

**対処：**
- 共通処理を1つの関数に統合する
- 統合した関数を呼び出す形にリファクタリングする
- 削除した関数の参照が残っていないか確認する

#### 4. 構造が整理されていない

**検出：**
- ディレクトリ構造を見て「どこに何があるか」が直感的にわかるか確認する
- 関連するコードが離れた場所に散らばっていないか確認する

**対処：**
- 関連するコードをまとめる
- ディレクトリ名・ファイル名がその中身を正確に表すようにする

### 実行手順

1. コードベース全体を俯瞰する（ファイル一覧・行数を確認）
2. 4パターンを順番にチェックし、問題箇所をリストアップする
3. ユーザーに問題箇所と修正案を提示する（y/n確認）
4. 承認を得てから修正を実施する
5. 修正後に全テストを実行して回帰がないことを確認する

### 原則
- 変更範囲を最小にする。必要な箇所だけ触る
- 各ステップでテストを通す
- 一度に全部直そうとしない。優先度の高いものから1つずつ進める

## 4. commit-flow — コミット・プッシュスキル

**いつ使うか：** 実装完了後、必ず動かすこと。「コミットしたい」「完成した」「プッシュしたい」と言ったときは必ずこのスキルを動かすこと。

**目的：** コードレビュー→修正→テスト→pre-commit→コミット→プッシュを一連のフローとして実行する。テストが通っていない状態・pre-commitが失敗した状態でコミットしない。

### Step 1: セルフレビュー

```bash
git diff
git diff --staged
```

確認項目：
- 変更が意図した範囲に収まっているか
- デバッグ用のprint・console.log・一時コードが残っていないか
- 自分の変更が生み出した未使用のimport・変数・関数がないか
- セキュリティ上の問題がないか（ハードコードされた認証情報等）

問題があれば修正する。

### Step 2: テスト実行

全テストを実行する。関連テストだけでなく全体を実行する。

```bash
# Python
uv run pytest

# Node.js
npm test
```

テストが通らない場合はコミットしない。原因を特定して修正する。

### Step 3: 静的解析

```bash
# Python
uv run ruff check .
uv run mypy .

# Node.js
npm run lint
npm run typecheck
```

エラー・warningを修正する。

### Step 4: pre-commit

```bash
git add <対象ファイル>
uv run pre-commit run
```

失敗した場合はコミットしない。自動修正されたファイルを確認し、再度 `git add` してから再実行する。

### Step 5: コミット

Conventional Commits形式を使う。

```bash
git commit -m "$(cat <<'EOF'
<type>: <subject>

<body（必要な場合）>
EOF
)"
```

**typeの選択：**
- `feat` — 新機能
- `fix` — バグ修正
- `refactor` — 動作を変えないリファクタリング
- `test` — テストの追加・修正
- `chore` — ビルド・設定等

**良いコミットメッセージの条件：**
- 1行目は50文字以内
- 「何をしたか」ではなく「なぜしたか」を書く
- 異なる種類の変更は別コミットに分ける

### Step 6: プッシュ

```bash
git push origin <branch>
```

### 禁止事項
- テストが通っていない状態でコミットしない
- pre-commitが失敗した状態でコミットしない
- `--no-verify` を使わない
- 異なる種類の変更を1コミットにまとめない
- `git add .` や `git add -A` を使わない（意図しないファイルが含まれる）

## 5. skill-improve — スキル改善メタスキル

**いつ使うか：** スキルが完了したタイミング。「スキルがうまく動かなかった」「改善したい」「振り返りたい」と感じたとき、またはスキルを一通り使い終えたあと。

**目的：** 直前に使ったスキルの振り返り・改善、概念のズレの記録、lessonsの更新を行う。スキルは使うたびに改善されることで価値が増す。

### Step 1: スキルの振り返り

**うまくいったか：**
- 期待通りの結果が出たか？
- スキルの手順は自然に流れたか？
- 省略・スキップしたステップはあったか？それは妥当だったか？

**うまくいかなかったか：**
- どのステップで詰まったか？
- なぜ詰まったか？（スキルの指示が曖昧 / 前提が違った / 想定外のケース）
- 次回同じ状況でどうすれば改善できるか？

### Step 2: 概念・用語のズレを記録する

「これをやって」と言ったときにAIの解釈が意図と違った場合、そのズレを記録する。

`alignment.md`（プロジェクトルートまたは `tasks/` 直下）に追記する：

```markdown
## [概念名]
- **ユーザーの意図**: 
- **AIの解釈（ズレていた場合）**: 
- **正しい定義**: 
- **記録日**: 
```

`alignment.md` が存在しない場合は新規作成する。

### Step 3: lessons.md を更新する

失敗・学びがあった場合は `tasks/lessons.md` に追記する。

```markdown
### [タイトル（キーワードを含む1行）]
- **症状**: 
- **原因**: 
- **対策**: 
```

「次回の自分が同じ失敗をしないために何を知っておけばよいか」を書く。

### Step 4: スキル定義を更新する

振り返りの結果、スキルの手順・表現・チェックリストに改善点があれば、スキルファイルを直接編集する。

- 曖昧だった指示を具体化する
- 不要なステップを削除する
- 見落としていたケースを追加する
- ファイルパス：`~/.claude/skills/<skill-name>/SKILL.md`

**判断基準：**
- 同じ問題が2回起きたら必ず修正する
- 小さくても改善する

### Step 5: architectスキルを走らせるべきか判断する

以下に該当する場合は `architect` スキルの実行を提案する：

- 実装が積み重なってきた（3回以上の実装サイクルを経た）
- ファイルが肥大化してきた感覚がある
- 似た処理を何度か書いた気がする

### 原則
- 「完璧な振り返り」を目指さない。気づいた点を素早く記録することが目的
- 振り返りに時間をかけすぎない（5〜10分で終える）
- 記録は未来の自分とAIへの手紙として書く
